## 1. 현재 앱 상태 요약

### 1.1 강점

1. **깔끔한 아키텍처**: 3페이지 SPA (Home -> Input -> Result)로 단순하고 이해하기 쉬움. 데이터/훅/계산/UI가 명확히 분리됨.
2. **계산 엔진 품질**: `calculator.ts`가 순수 함수 기반으로 설계됨. 일손익 -> 월손익 -> 투자회수 -> DCF까지 논리적 파이프라인.
3. **Payback 계산의 체감식 이자**: `calcPayback()` (calculator.ts:87-123)에서 매월 잔여 대출금 기준 이자를 계산하여 현실적.
4. **Supabase 이중화**: `supabase.ts`에서 env 변수 없으면 로컬 데이터로 자동 fallback. 오프라인에서도 동작.
5. **사용자 조정 가능**: ResultPage에서 슬라이더로 고객수/객단가/임대료/할인율/성장률을 실시간 조정 가능.
6. **디자인 토큰 체계**: `index.css`에 CSS 변수로 색상/간격/폰트 일원화.

### 1.2 약점

1. **테스트 부재**: 프로젝트 메모리에 57개 테스트 언급이 있으나, 현재 `src/` 하위에 `.test.*` 또는 `.spec.*` 파일이 하나도 없음. 재설계 시 회귀 방지 수단이 전무.
2. **costItems 데이터 미활용**: 89개 비용 항목이 있으나 실제 계산에는 전혀 사용되지 않음. `PnLDisplay.tsx:45`에서 표시용 힌트로만 사용. 데이터 유지보수 비용만 발생.
3. **업종별 차이 무시**: 편의점(24시간=30일), 학원(월 수강료 모델)이 모두 26일 영업일, 일일 객단가 모델로 강제됨 (`calculator.ts:7`).
4. **deposit_per_sqm 미사용**: `rentGuide.ts`에 보증금 데이터가 있으나 어디에서도 사용되지 않음 (types/index.ts:50).
5. **단일 인건비 구조**: `labor_cost_monthly_per_person`이 모든 직원에게 동일 적용. 사장(무급)/아르바이트/정규직 구분 없음.
6. **라우터 부재**: `App.tsx`에서 `useState`로 직접 페이지 관리. 브라우저 뒤로가기/딥링크 불가.
7. **Storybook 유명무실**: `package.json`에 storybook이 설치되어 있으나 실제 컴포넌트 story 없음.

## 2. 로직 허점 및 버그

### 2.1 학원(교습소) 매출 계산 구조적 오류 [심각도: 높음]

- **위치**: `calculator.ts:26-31` (calcDailyPnL), `businessTypes.ts:13` (학원 데이터)
- **문제**: 학원의 `avg_ticket_price`가 200,000원(월 수강료)인데, 계산식이 `ticketPrice x dailyCustomers x 26일`을 적용
- **결과**: 중규모 학원 매출 = 200,000 x 15 x 26 = **7,800만원/월** (실제 월수강료 기반 매출은 200,000 x 15 = **300만원/월**)
- **영향**: 학원 업종의 모든 계산(손익/투자회수/DCF)이 **26배 과대 산출**
- **근본 원인**: 모든 업종을 "일회성 결제 x 일일 고객수 x 영업일수" 모델로 강제. 구독/월납 모델 미지원.

### 2.2 헬스장(PT샵) 유사 문제 [심각도: 중간]

- **위치**: `businessTypes.ts:15`
- **문제**: `avg_ticket_price: 80,000`이 PT 1회 가격인지 월 회비인지 불명확. 만약 월 회비라면 학원과 동일 문제.
- **현재 계산**: 80,000 x 25 x 26 = **5,200만원/월** (현실성 의문)

### 2.3 ResultPage 슬라이더 재계산 타이밍 문제 [심각도: 중간]

- **위치**: `ResultPage.tsx:23-29`
- **문제**: `handleOverrideAndRecalc`에서 `onOverride(key, value)` 후 즉시 `onRecalculate()`를 호출. 하지만 `onOverride`는 React의 `setOverrides`를 호출하므로 상태 업데이트가 비동기. `onRecalculate` 시점에 아직 이전 inputs로 계산될 수 있음.
- **실제 동작**: `useSimulator.ts:65-73`의 `calculate`가 `useMemo`로 생성된 `inputs`를 사용. `inputs`는 `overrides` 상태에 의존하므로, override 설정 직후 calculate 호출 시 **이전 렌더의 inputs로 계산**됨.
- **증상**: 슬라이더를 움직이면 결과가 "한 틱 뒤처져" 보일 수 있음. 두 번째 조정부터 이전 값의 결과가 보임.
- **원인**: React의 batched state update. `setOverrides` -> `useMemo(inputs)` -> `calculate(inputs)` 순서가 같은 렌더 사이클에서 완료되지 않음.

### 2.4 DCF 성장률 >= 할인율 시 음수 가치 [심각도: 낮음]

- **위치**: `calculator.ts:133-136`
- **문제**: `denominator === 0`일 때만 null 반환. 하지만 `growth_rate > discount_rate`일 때 denominator가 음수가 되어 **음수 사업가치**가 산출됨.
- **사용자 시나리오**: 성장률 슬라이더를 할인율 이상으로 올리면 "-2억원" 같은 무의미한 결과 표시.
- **수정**: `denominator <= 0` 조건으로 변경 필요.

### 2.5 세금 계산 시 인적공제 미반영 [심각도: 낮음-참고]

- **위치**: `tax.ts:18-22`
- **현재**: 과세표준 = 월 세전소득 x 12 로 바로 세율 적용
- **실제**: 기본공제 150만원, 표준세액공제 등이 적용되어야 함
- **영향**: 세금이 약 10~30만원/월 과대 산출 (보수적이므로 의도적일 수 있음, APPENDIX에도 한계로 명시됨)

### 2.6 Payback에서 operating_profit 이중 계산 [심각도: 낮음]

- **위치**: `calculator.ts:98-99`
- **문제**: `calcPayback` 내부에서 `calcMonthlyPnL(inputs)`를 호출하여 `gross_profit`과 `sg_and_a`를 가져온 후 `operating_profit = gross_profit - sg_and_a`를 다시 계산. `calcMonthlyPnL`의 결과에 이미 `operating_profit`이 있으므로 중복. 기능상 동일하나 불필요한 재계산.

### 2.7 면적(scaleSqm) 하드코딩 [심각도: 낮음]

- **위치**: `InputPage.tsx:72`
- **문제**: `scaleSqm = scale === 'small' ? 33 : scale === 'large' ? 66 : 50` 이 InputPage에 하드코딩. 이 값이 `BusinessType`이나 `types`에 정의되어 있지 않아 업종별로 다른 면적을 적용할 수 없음.

  - `costItems.ts`의 `business_type_id` 외래키 전면 재매핑 필요
  - Supabase `business_types` 테이블 마이그레이션 필요

#### B. 초기투자 분해 -> CapitalStructure 및 Payback 재설계

  - `CapitalStructure` 타입 확장 필요

  - 보증금은 회수 가능하므로 payback 계산에서 차감해야 하는지 결정 필요
  - InputPage의 슬라이더가 단일 -> 다중으로 확장

#### C. 프랜차이즈 비용 -> BusinessType 분기 필수

- 프랜차이즈 업종은 가맹비/로열티/광고분담금이 추가됨
- **현재**: `costItems.ts`에 "프랜차이즈 가맹비"가 `is_initial_cost: true`로 있으나 계산에 미사용
- **변경 필요**: 프랜차이즈 여부를 BusinessType에 필드 추가하거나, 별도 타입으로 분리

#### D. UI 플로우 재설계 -> useSimulator 상태 모델 전면 교체

  - `App.tsx`의 `Page = 'home' | 'input' | 'result'` -> 스텝 기반으로 변경
  - `useSimulator`의 `inputs` 구성이 스텝별로 점진적 빌딩 필요
  - 현재는 `setBusinessType` 시 자동으로 capital 초기화 (`useSimulator.ts:34-42`) -> 스텝 분리 시 이 로직 재배치

#### E. 가이드라인 데이터 -> 새로운 데이터 레이어 필요

- 입력값 가이드라인(예: "이 업종의 현실적 고객수 범위는 X~Y명")은 현재 `BusinessType`의 `avg_daily_customers_small/medium/large`에 암묵적으로 존재
- 별도의 가이드라인 데이터 구조를 만들면:
  - 새 타입 정의 필요
  - 새 데이터 파일 + Supabase 테이블 필요
  - UI에서 가이드라인 표시 컴포넌트 필요

## 4. 구현 순서 제안 (의존성 그래프)

### 4.1 의존성 그래프

```
Phase 0 (선행)
  [0-A] 테스트 복원/작성 ─────────────────────────────────┐
  [0-B] 학원/헬스장 매출 모델 버그 수정                      │
  [0-C] DCF denominator <= 0 수정                         │
  [0-D] ResultPage 재계산 타이밍 수정                       │
                                                          │
Phase 1 (기반)                                             │
  [1-A] BusinessType 타입 확장 ◄──── 업종세분화 리서치       │
       (프랜차이즈 여부, 매출모델 타입, 업종 면적)            │
  [1-B] CapitalStructure 타입 확장 ◄── 초기투자분해 리서치   │

  [2-A] businessTypes.ts 데이터 확장/분할                    │
  [2-B] costItems.ts 재매핑                                │
  [2-C] 가이드라인 데이터 파일 생성                           │
  [2-D] Supabase 스키마 마이그레이션                         │
                                                          │
Phase 3 (계산 엔진)  ← Phase 2 완료 후                      │
  [3-A] calculator.ts: 매출 모델 분기 (일회성/구독/혼합)      │
  [3-B] calculator.ts: 프랜차이즈 비용 반영 (로열티, 광고비)   │

Phase 4 (UI)  ← Phase 3 완료 후                            │
  [4-A] UI 플로우 재설계 (멀티스텝/위저드) ◄── UI리서치       │
  [4-B] 가이드라인 표시 UI                                   │
  [4-C] 초기투자 항목별 입력 UI                              │
  [4-D] 프랜차이즈 비용 입력 UI                              │
                                                          │
Phase 5 (검증)  ← Phase 4 완료 후                           │
  [5-A] 전체 테스트 업데이트 ◄──────────────────────────────┘
  [5-B] Supabase 연동 테스트
  [5-C] 모바일 성능 테스트
  [5-D] 엣지 케이스 검증
```

### 4.2 병렬 가능한 작업

| 병렬 그룹 | 작업들 | 이유 |
|-----------|--------|------|
| Phase 0 | 0-A, 0-B, 0-C, 0-D | 모두 독립적 버그 수정 |
| Phase 1 | 1-A, 1-B, 1-C | 타입 정의는 서로 독립 |
| Phase 2 | 2-A + 2-B, 2-C | 데이터 파일은 타입만 맞으면 병렬 가능 |
| Phase 4 | 4-B, 4-C, 4-D | 4-A(플로우) 완료 후 개별 UI는 병렬 가능 |

### 4.3 순차 필수 작업

```
타입 정의(Phase 1) -> 데이터(Phase 2) -> 계산 엔진(Phase 3) -> UI(Phase 4) -> 테스트(Phase 5)
```

이 순서를 어기면 타입 불일치로 컴파일 에러가 연쇄 발생함.

## 5. 누락된 고려사항

### 5.1 지역 선택이 새 플로우에서 빠질 위험

- **현재**: InputPage에서 RegionSelector로 지역 선택 -> 월 임대료 자동 계산 (InputPage.tsx:66-75)
- **위험**: UI 플로우 재설계 시 스텝 분리하면서 지역 선택이 누락될 가능성
- **대안**: 지역 선택을 독립 스텝으로 확보하거나, 임대료 입력의 "가이드 모드"로 통합

### 5.2 테스트 코드 전면 작성 필요

- 현재 테스트 파일이 **0개**. 프로젝트 메모리의 "57 tests" 기록과 불일치.
- 재설계 전에 최소한 `calculator.ts`와 `tax.ts`의 단위 테스트를 복원/작성해야 함
- **예상 공수**: calculator.ts 테스트 30개 + tax.ts 테스트 10개 + hooks 테스트 15개 = 약 55개, 2-3일

### 5.3 Supabase 스키마 마이그레이션

- BusinessType 확장 -> `business_types` 테이블 ALTER 필요
- 초기투자 분해 -> 새 `investment_items` 테이블 또는 기존 테이블 컬럼 추가
- 가이드라인 -> 새 `input_guidelines` 테이블
- costItems 재매핑 -> `cost_items` 테이블 데이터 마이그레이션
- **주의**: Supabase 마이그레이션 스크립트가 현재 없음. `supabase/migrations/` 디렉터리 필요.

### 5.4 모바일 성능

- **현재**: 3페이지로 가벼움. Recharts가 가장 큰 번들 (chunk size warning 기존 인지).
- **위험**: 스텝이 많아지면:
  - 각 스텝마다 SliderInput 다수 렌더링 -> 리렌더 증가
  - 가이드라인 데이터 + 프랜차이즈 데이터 추가 fetch -> 초기 로딩 지연
  - 해결: React.lazy + 스텝별 코드 스플리팅, 또는 데이터 프리페칭

### 5.5 공과금/배달앱 수수료 미반영

- `costItems.ts`에 공과금(utilities)과 배달앱 수수료(marketing)가 데이터로 존재하나, `calcMonthlyPnL`에서는 `misc_fixed_cost_monthly` 하나로 퉁침.
- 재설계 시 이 비용들을 계산에 반영할지 결정 필요.

### 5.6 부가가치세(VAT) 미반영

- 외식업은 매출의 10% VAT 납부 의무 (간이과세 vs 일반과세)
- 현재 계산에 VAT가 전혀 없음. 소상공인 시뮬레이터로서 중대한 누락.
- 간이과세자(연 8,000만원 미만) vs 일반과세자 분기 필요.

### 5.7 감가상각비 미반영

- 초기투자의 인테리어/설비는 5년 정액법 감가상각이 세무상 비용으로 인정됨
- 세금 계산 시 감가상각비를 비용으로 차감하면 실효세율이 낮아짐
- 초기투자 분해 리서치와 연계하여 반영 검토 필요

### 5.8 광고 SDK 통합 준비

- 토스 미니앱 내 광고 SDK 통합이 필요할 경우:
  - 토스 광고 정책 확인 필요 (미니앱 내 광고 허용 여부)
  - 광고 컴포넌트 렌더링 위치 설계 (ResultPage 하단 등)
  - 현재 아키텍처에서는 추가 영향 없음 (독립 모듈로 통합 가능)

### 5.9 data_sources 필드 미활용

- `BusinessType.data_sources: string[]` (types/index.ts:24)가 UI에 노출되지 않음
- 앱 내 "출처 보기" 기능이나 부록 페이지에서 활용 가능
- APPENDIX-DATA-SOURCES.md 문서는 존재하나 인앱 부록은 미구현

## 6. 리스크 평가

### 6.1 공수 예측

| Phase | 작업 | 예상 공수 | 난이도 |
|-------|------|-----------|--------|
| Phase 0 | 버그 수정 + 테스트 복원 | 3-4일 | 낮음 |
| Phase 1 | 타입 확장 | 1-2일 | 낮음 |
| Phase 2 | 데이터 확장 + Supabase 마이그레이션 | 3-5일 | 중간 |
| Phase 3 | 계산 엔진 재설계 | 5-7일 | 높음 |
| Phase 4 | UI 플로우 재설계 | 7-10일 | 높음 |
| Phase 5 | 검증 + 테스트 업데이트 | 3-5일 | 중간 |
| **합계** | | **22-33일** | |

### 6.2 기존 기능 깨질 위험

| 위험 영역 | 확률 | 영향 | 대응 |
|-----------|------|------|------|
| BusinessType 타입 변경 -> 전체 컴파일 에러 | 높음 | 높음 | TypeScript strict로 컴파일 에러 조기 발견 |
| Payback 계산 변경 -> 결과 수치 변동 | 높음 | 중간 | 기존 결과 스냅샷 테스트 작성 후 비교 |
| UI 플로우 변경 -> 기존 사용 패턴 파괴 | 높음 | 중간 | 기존 3페이지 플로우를 fallback으로 유지 가능 |
| Supabase 스키마 변경 -> 기존 데이터 손실 | 중간 | 높음 | 마이그레이션 스크립트 + 백업 필수 |
| Recharts 추가 차트 -> 번들 사이즈 증가 | 낮음 | 낮음 | 이미 Recharts 사용 중, 추가 영향 미미 |

### 6.3 데이터 마이그레이션

- **로컬 데이터**: `src/data/` 파일 직접 수정으로 해결 (위험 낮음)
- **Supabase 데이터**: 현재 env 미설정 상태로 추정 (실제 사용 여부 확인 필요)
  - 사용 중이면: ALTER TABLE + INSERT 마이그레이션 필수
  - 미사용이면: 스키마만 맞춰두면 됨
- **하위 호환성**: `supabase.ts`의 `as BusinessType[]` 캐스팅 (supabase.ts:29)이 타입 변경 시 런타임 에러 유발 가능. Zod 등 런타임 검증 추가 권장.

### 6.4 점진적 배포 vs 빅뱅 배포

| 전략 | 장점 | 단점 |
|------|------|------|
| 빅뱅 (한번에 전부) | 일관성, 중간 불일치 없음 | 롤백 어려움, 테스트 범위 넓음 |
| 점진적 (Phase별 배포) | 위험 분산, 피드백 수집 가능 | 데이터 구조 변경은 한번에 해야 함 |
| **권장**: Phase 0-2를 한번에, Phase 3-4를 단계별 | | |

## 7. 사용자에게 질문할 사항

### 7.1 비즈니스 결정 필요

1. **학원/헬스장 매출 모델**: 월 수강료 모델(구독형)을 별도 계산 로직으로 분기할 것인가, 아니면 객단가를 일회성 기준으로 수정할 것인가?

2. **프랜차이즈 vs 개인 창업**: 같은 업종 내에서 프랜차이즈/개인을 완전히 다른 업종으로 나눌 것인가, 아니면 토글/옵션으로 처리할 것인가? (데이터 구조에 큰 영향)

3. **부가가치세(VAT)**: 간이과세/일반과세 분기를 포함할 것인가? 포함하면 계산 복잡도가 크게 증가.

4. **costItems 89개**: 현재 계산에 미사용. 재설계 시 이 데이터를 계산에 반영할 것인가, 아니면 참고 표시로 유지할 것인가?

5. **보증금(deposit_per_sqm)**: 초기투자에 보증금을 포함시킬 것인가? 보증금은 회수 가능하므로 payback 계산이 달라짐.

### 7.2 기술 결정 필요

6. **Supabase 현재 사용 여부**: env 변수가 설정되어 있는가? 프로덕션에서 Supabase를 실제 사용 중인가?

7. **토스 미니앱 출시 시기**: TDS 전환과 재설계를 동시에 할 것인가, 먼저 기능 재설계 후 나중에 TDS 전환할 것인가?

8. **테스트 전략**: 57개 테스트가 어떻게 사라졌는가? 복원 가능한가? 새로 작성할 것인가?

9. **라우터 도입**: 멀티스텝 위저드 전환 시 React Router를 도입할 것인가? 토스 미니앱 WebView에서 라우터가 정상 동작하는지 확인 필요.

10. **감가상각비/인적공제**: 세금 계산의 정확도를 높일 것인가? 높이면 사용자 입력 항목이 늘어남 (부양가족 수, 자산 내역 등).
